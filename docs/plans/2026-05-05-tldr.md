# Next phase — TL;DR

* **Dashboard with insights on Vercel w/ Next.js.** Day view, BG heatmap, TIR trends, search/filter, comparisons. Built in phases: Phase A views work today against current data; richer views switch on as detection improves.

* **Supabase Postgres for storage + auth.** CGM, pump, alert state, config. ~36MB/year against 500MB free tier (~14 years headroom).

* **pydexcom live feed on Vercel w/ 5-min cron.** External trigger (cron-job.org) hits a Vercel function. The live alert path's only data dependency.

* **Telegram alert pipeline (deterministic, pre-templated).** No LLM in the critical path; reliability and auditability matter too much for alerts to depend on an external API.

* **Telegram pull commands via webhook + DeepSeek LLM.** Time-scoped summaries, explain-a-moment, endo prep, settings flagging. User-initiated, so an LLM outage just means "try again."

* **Detection engine rebuilt from first principles.** Three output layers (events for alerts, episodes for narrative, patterns for multi-day insight). CGM-only meal-shape detector calibrated against historical Tandem-labeled rises. Pure DataFrame-in / DataFrame-out, storage-agnostic. *we must ideate and test potential strategies here - we will work together on this*

* **Tandem nightly sync via GitHub Actions, off the live path.** Live missed-meal detection works on rise shape alone (80-90% reliable); Tandem data feeds dashboard context, LLM payload, calibration, and pattern analysis. GitHub Actions runs the nightly workflow (free, 6-hr timeout); Vercel cron is the fallback if Actions can't auth against Tandem in CI.

* **Scheduled LLM pushes (morning briefing, weekly digest).** Same pull-command machinery, pushed instead of triggered. *ideation still in progress, do not implement witout discussing*

* **Core / shell architecture for OSS-friendliness.** Core library (ingestion, enrichment, detection) imports nothing about Supabase, Vercel, or any storage backend. Personal stack (Next.js + Vercel + Supabase + GitHub Actions) and a future open-source local stack (Streamlit + parquet/SQLite, single-YAML config, optional Telegram long-polling) are both thin shells over the same core. Storage hides behind a Protocol with `ParquetStorage` and `SupabaseStorage` implementations. **VERY IMPORTANT**

* **Operational guardrails for free tiers.** Vercel Hobby cron is daily-only (live loop uses external cron-job.org). Supabase free-tier egress is 5 GB/month, so dashboard heavy reads (heatmap, TIR trends) aggregate Postgres-side, not client-side. Vercel functions use Supabase's transaction-mode pooler URL; SupabaseStorage enforces open-do-close lifecycle. Idle-in-transaction timeout set at Postgres level as backstop. No realistic concurrent-connection risk; the binding concern is leak prevention.
